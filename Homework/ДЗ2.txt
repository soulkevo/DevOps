2. 	Я працюю в команді, яка підтримує та розвиває програмний продукт. Робота організована за принципами Agile, однак має свої особливості. Робочий процес починається з отримання задачі в Jira. Після ознайомлення з вимогами виконується аналіз проблеми або нового функціоналу. Для перевірки роботи сервісів використовуються різні інструменти, зокрема SQL для аналізу даних, Postman для тестування API, Kibana для перегляду логів та ArgoCD для контролю стану застосунків. Під час виконання задачі активно відбувається взаємодія з розробниками, QA-інженерами та іншими членами команди. Якщо виникають питання або виявляються проблеми, вони обговорюються у внутрішніх чатах або під час командних зустрічей.
	Мій робочий процес відрізняється від класичного Agile тим, що основна діяльність команди пов'язана не з розробкою нового функціоналу, а з 
підтримкою та супроводом існуючих сервісів. Значна частина часу витрачається на аналіз інцидентів, дослідження логів, взаємодію з партнерами та іншими командами, а також на вирішення проблем, що виникають у продуктивному середовищі. Для роботи використовуються Jira для управління задачами, Kibana для аналізу логів, Grafana для моніторингу сервісів, Postman для перевірки API та Atlassian Confluence для роботи з документацією. Про критичні інциденти команда отримує сповіщення через Slack та PagerDuty. На відміну від класичного Agile, частина задач виникає непередбачувано та потребує негайної реакції. Тому пріоритети можуть змінюватися залежно від поточного стану систем та критичності інцидентів.
	Для покращення я б запропонував доцільно розширити автоматизацію рутинних перевірок та post-deployment checks. Наразі значна частина перевірок 
виконується вручну Support Engineer. Автоматичні сценарії перевірки API, доступності сервісів та ключових бізнес-потоків після розгортання дозволили б швидше виявляти проблеми та зменшити навантаження на команду підтримки.
